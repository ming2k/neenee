# 文档更新工作流

按步骤更新文档，确保代码变更与文档保持一致。

## 前置阅读

开始之前，请先确认：
1. 仓库采用的文档结构约定——见 [Adoption](adoption.md)。
2. 内容应该放在哪里——见 [Routing](routing.md)。
3. 文档应该遵循的写作风格——见 [Writing Style](style-guide.md)。

## 步骤化更新流程

### Step 1: 变更影响评估（变更前）

- [ ] 确认本次代码变更的类型：
  - 公共 API、CLI 参数、配置项、Schema 字段变更
  - 新增用户可发现功能
  - 安装、构建或运行步骤变更
  - 开发环境或测试命令变更
  - 架构决策变更
  - 用户可见行为变更
  - 纯内部重构（无文档影响）
- [ ] 根据变更类型，确定需要更新的文档区域（见下方映射表）

### Step 2: 确定文档目标位置（变更前）

| 变更类型 | 目标文档区域 |
|---------|-------------|
| 公共 API、CLI、配置、Schema | `docs/reference/` |
| 新增用户功能 | `docs/how-to/` |
| 安装、构建、运行步骤 | `README.md` + 相关教程 |
| 开发环境、测试命令 | `docs/dev/` |
| 架构决策 | `docs/adr/`（新增 ADR） |
| 用户可见行为 | `CHANGELOG.md`（Unreleased 条目） |
| 纯内部重构 | 无需文档更新 |

如果仓库未使用上述文档表面，请改用 [Adoption](adoption.md) 中记录的等效位置。若等效位置不存在，请在 PR 中说明，而非临时发明一个一次性位置。

### Step 3: 文档编写与更新（变更中）

- [ ] 在对应目录创建或更新文档文件
- [ ] 遵循现有文档格式和风格（参考 `docs/dev/documentation/style-guide.md`）
- [ ] 如新增页面，更新 `docs/index.md` 和对应章节的 `index.md` 导航
- [ ] 如新增 ADR，使用 `docs/adr/template.md` 模板

### Step 4: 交叉验证（变更后）

- [ ] 检查所有相关 `index.md` 链接是否有效
- [ ] 检查文档中引用的代码片段、命令、路径是否与实际代码一致
- [ ] 运行文档格式检查（如有）

### Step 5: PR 说明（提交时）

- [ ] 在 PR 描述中明确说明：文档已更新 / 未更新及原因
- [ ] 引用本次变更对应的文档更新清单条目

如果 PR 触及了具有对应文档表面的代码，必须在 PR 描述中说明文档是否已更新，或解释为何不需要更新。

## 应用示例

**场景：新增一个 Provider**

1. **评估**：属于"新增用户可发现功能" + "公共 API 变更"
2. **目标**：`docs/how-to/add-a-provider.md`（更新步骤） + `docs/reference/providers.md`（更新参考列表）
3. **编写**：更新 how-to 指南，补充 reference 条目
4. **验证**：检查 `docs/reference/index.md` 和 `docs/how-to/index.md` 导航链接
5. **PR 说明**：注明更新了 provider 添加指南和参考文档

## 工具支持

- 使用 `grep` 或 `task` 搜索文档中引用的代码符号，确认一致性
- 使用 `read_file` 查看现有文档模板和风格指南
- 变更提交前，使用 `list_dir` 确认文档目录结构未遗漏
