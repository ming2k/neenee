# 0022. Session-level AI title

- **Status:** Accepted
- **Date:** 2026-06-24

## Context

A session in this project is identified to the user by `SessionSummary::overview`
(`neenee-store/src/session.rs`), which `session_overview()` derives as the first
user message truncated to 64 chars (falling back to the active pursuit, then
`(empty session)`). That is the same mechanical "first line" heuristic codex
uses locally (`summarize_for_label` in `codex-rs/external-agent-sessions`) and
the fallback path opencode and claude-code take when their AI title has not run.

The first-line heuristic is cheap but weak: a verbose opening prompt
(`"hey, so i was thinking about whether we could maybe refactor the auth
middleware to..."`) becomes an unrecognizable picker row. opencode and
claude-code both spend one cheap-model call to turn the first real user
message into a 3–7 word title; codex defers AI titles to its cloud backend.
neenee has no equivalent today.

We already have the structural precedent for an on-demand sub-agent that reads
a transcript snapshot and returns a short structured result: session review
(ADR-0016, revised by ADR-0018). It defines a `REVIEW` profile in
`neenee-core/src/subagent.rs`, a domain trait in
`neenee-core/src/session_review.rs`, and a runner in
`neenee-agent/src/session_review.rs` that spawns a bounded child `Agent`. A
title generator is the same shape with a stricter profile (no tools needed) and
a simpler runner (one chat call, no tool loop).

Two scopes were considered: a title for the **whole session**, and an intent
**subtitle per user turn**. This ADR addresses only the session-level title.
Per-turn subtitles are deferred — see Alternatives.

## Decision

1. **Add a session-level AI title** generated by a sub-agent, stored durably on
   the session.

2. **Trigger policy (generate-once-then-stable, the cross-tool best practice):**
   - **First turn, automatic.** After a turn completes successfully and its
     history is persisted (`execute_turn`, `neenee-agent/src/orchestration.rs`,
     just past the `session.replace_messages` call), if no title exists and the
     transcript holds exactly one real user message, spawn the title sub-agent
     in the background and persist the result. The "exactly one real user
     message" guard mirrors opencode's `ensureTitle` condition and means a
     resumed multi-turn session does not silently retro-title itself.
   - **On demand.** A `/title` command regenerates the AI title from the current
     transcript.
   - **Stable afterwards.** No cadence-based re-generation on later turns.
     opencode and claude-code both lock the title after first generation; an
     auto-refresh every turn would burn a call per turn and make the picker row
     churn.
   - **Manual lock wins.** `/title <text>` sets a manual title and marks it
     locked; automatic and on-demand AI generation never overwrite a locked
     title. This matches opencode's `setTitle` vs `isDefaultTitle` distinction
     and codex's `distinct_thread_metadata_title` rule.

3. **Display precedence:** manual title → AI title → existing first-user-message
   `session_overview()` fallback. The fallback path is unchanged, so sessions
   written before this ADR (and sessions whose title call fails) render exactly
   as today.

4. **New `TITLE` sub-agent profile** in `neenee-core/src/subagent.rs`, alongside
   `EXPLORE`/`VERIFY`/`REVIEW`. It is read-only and non-interactive like
   `REVIEW`, but its task needs no tools at all, so the runner does a single
   `provider.chat()` with a title-oriented system prompt rather than driving a
   tool loop. Defining it as a profile (not an ad-hoc call) keeps the
   capability-axis vocabulary the single source of truth for what a bounded
   sub-agent may do, per ADR-0011/0012.

5. **Model selection is the sub-agent's concern, not special-cased here.** The
   runner uses the session's own provider (`self.provider.clone()`), exactly as
   `run_session_review` does. opencode routes titles through a `getSmallModel`
   channel and claude-code through Haiku; neenee's catalog has no "small model"
   concept, and introducing one is out of scope. A user who wants a cheaper
   title model gets it the same way they would for any sub-agent — by the
   sub-agent selecting its provider — not via a title-specific config key.

6. **Persistence schema bump (1 → 2).** `SessionData` gains:
   - `title: Option<String>` — the current effective AI title, if any.
   - `title_manual: bool` (serde default `false`) — whether the title was set
     by `/title <text>` and must not be overwritten by AI generation.

   A new `SessionEvent::TitleSet { title, manual }` makes title changes durable
   in the per-session event log. `migrate_session_data` gains a `schema_version
   < 2` guard that is a structural no-op (legacy files load with `title = None`,
   `title_manual = false`, so they render via the unchanged fallback until they
   get a fresh first turn or a `/title` call).

## Alternatives considered

- **Per-turn intent subtitles (turn-level), now.** Rejected for this ADR. It is
  a distinct, heavier feature: it costs one model call per turn (N calls for N
  turns, where session-level costs one), and `Message` has no field to carry a
  subtitle, so it touches the message schema rather than just the session
  envelope. Neither opencode nor claude-code generates per-turn labels. It is
  closer to an outline/TOC feature and should be scoped on its own.

- **Auto-regenerate the title on a cadence or on topic shift.** Rejected. Every
  cross-tool precedent generates once and locks; periodic re-generation spends a
  call per turn and makes the picker row change under the user. The on-demand
  `/title` is the escape hatch when the topic genuinely moved.

- **A separate "small/cheap" model channel for titles.** Rejected as a
   precondition. It would require a new catalog concept (neenee's picker is
   per-provider, not tiered) and a new config surface. Using the session's
   provider keeps the feature self-contained; a dedicated cheap channel can be
   layered on later without changing this design.

- **Drive a full tool-loop sub-agent (like `REVIEW`).** Rejected. Title
   generation is pure text-in/text-out; the profile admits no tools, so a tool
   loop could never advance. A single `provider.chat()` is simpler, cheaper, and
   still framed as the `TITLE` profile for consistency with the
   capability-axis vocabulary.

- **No on-demand path; first-turn auto only.** Rejected. Without `/title` a
   user stuck with a bad first-turn title (or a resumed session that predates
   this ADR and never auto-titles) has no recourse short of editing the session
   file. The manual lock + regenerate command is the standard escape hatch.

## Consequences

- **Positive.** Session picker rows become recognizable; the fallback heuristic
  is preserved as the failure/legacy path; the trigger policy matches the
  cross-tool best practice and spends at most one call per session plus optional
  on-demand calls.

- **Schema migration.** `CURRENT_SCHEMA_VERSION` rises to 2. Legacy snapshots
  load with no title and render via `session_overview()` until they gain a title
  through a first turn or `/title`. The migration is a structural no-op guarded
  on `schema_version < 2`, following the lazy-migration pattern already in
  `migrate_session_data`.

- **Cost.** One extra provider call on the first turn of each new session, plus
  one per explicit `/title`. The call is non-blocking (spawned after the turn
  persists) so it does not delay the response.

- **Concurrency.** Title persistence goes through the existing `SessionStore`
  mutex (`set_title`), so a title write from the background task cannot race a
  concurrent `replace_messages`. The generation-counter / cancellation machinery
  in `execute_turn` is unaffected — title generation runs after the turn's
  owned task completes.
